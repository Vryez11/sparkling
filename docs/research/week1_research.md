### 1. MVP란 무엇인가

#### 조사 내용

> MVP (Minimum Viable Product)
> 
> 스타트업 제품 개발 시 시장 반응을 빠르게 검증하기 위해 고객의 핵심 니즈를 충족하는
> 최소한의 기능만을 담아 출시하는 **최소 기능 제품**을 뜻한다.

```text
1. 완성도 낮은 MVP
    최소한의 기능으로 실제 제품의 형태만을 모방하여 구현된 MVP이다.
        - 랜딩 페이지
        - cardboard mockup

2. 완성도 높은 MVP
    실제 제품에 가깝게 구현된 MVP이다.
        - 고객의 솔루션에 대한 유효성을 검증하고 피드백을 모으는데 주로 사용된다.
     
현재 저희가 필수 기능으로 만들려는 완성도 높은 MVP로 생각했다.

이를 통해 우리가 얻고자하는 것은 사용자가 실제로 제품을 어떻게 사용하는지, 
어떤 기능에 머무르는지 행동 데이터를 수집하고자 한다.
```

### 2. 작은 서비스에서 범위를 자르는 기준

#### 여러 기준

> **핵심 가치(MVP) 중심 분리**
>
> 서비스가 해결하고자 하는 단 하나의 주요 문제만 해결하도록 범위를 좁히는 방식

> **사용자 행동 흐름(User Flow) 분리**
> 
> 가입부터 결과 확인까지, 사용자가 서비스를 이용하는 최소한의 필수 경로만 남기고 나머지는 수동 처리로 대체하는 방식

> **비즈니스 및 도메인 분리**
> 
> 유사한 기능이나 데이터 성격에 따라 서비스 경계를 나누는 방식, MSA 서비스 분리 원칙을 참고하여 독집적으로 수행 가능한 최소 단위 도메인을 찾는다.

> **데이터베이스 트랜잭션 단위 분리**
> 
> 여러 기능이 하나의 데이터베이스에 강하게 얽혀 있다면 범위를 나누기 어렵다.
> 데이터 변경이 독립적으로 일어나는 기능을 기준으로 자르는 방식

### 3. 넣으면 프로젝트가 급격히 커지는 기능은?

#### 공통 특징

기능 하나를 넣었을 뿐인데 코드와 인프라가 몇 배로 커지는 경우가 있다.
아래 특징 중 하나라도 해당하면 프로젝트를 급격히 키우는 기능으로 볼 수 있다.

> **비동기 / 실시간성**
>
> 요청-응답 한 사이클로 끝나지 않는 기능이다. 메시지 큐, 웹소켓, 폴링, 별도 워커가 필요해진다.

> **팬아웃(fan-out)**
>
> 한 번의 쓰기가 여러 사용자에게 전파되는 기능이다. 쓰기량이 폭증하고 별도 저장 구조가 필요해진다.

> **별도 저장소 / 인프라 요구**
>
> RDB(MySQL)만으로 해결되지 않는 기능이다. 검색엔진, 캐시, 오브젝트 스토리지 등이 추가된다.

> **집계 / 랭킹**
>
> 실시간 정합성과 배치 중 무엇을 택할지 결정이 필요한 기능이다. 스케줄러, 집계 테이블, 캐시가 따라온다.

> **재귀 / 트리 구조**
>
> 데이터가 스스로를 참조하는 기능이다. 깊이 제한, 정렬, N+1 조회 문제가 함께 발생한다.

#### 선택지

`scope.md` 기준으로 넣으면 프로젝트가 커지는 기능을 위 특징에 대입해 보았다.

```text
1. 알림        → 비동기 + 팬아웃
   거의 모든 기능(좋아요/댓글/팔로우)에 훅이 박혀야 하고,
   실시간이면 웹소켓/FCM, 아니면 알림 테이블 + 읽음 처리까지 필요하다.

2. DM(채팅)    → 비동기(실시간) + 상태 관리
   웹소켓 + 대화방 모델링 + 읽음 상태까지 필요하다.

3. 인기글      → 집계 / 랭킹
   실시간 카운트냐 배치냐 결정이 필요하고 스케줄러 + 집계 테이블 + 캐시가 붙는다.

4. 검색        → 별도 저장소 요구
   RDB LIKE로 시작하면 곧 한계에 부딪혀 형태소 분석/Elasticsearch로 번진다.

5. 팔로잉 피드  → 팬아웃
   팔로워가 많아지면 조회 시 join(pull)이냐 쓰기 시 전파(push)냐 아키텍처 결정이 필요하다.

6. 대댓글      → 재귀 / 트리 구조
   댓글이 트리가 되면서 조회/정렬/깊이 제한 로직이 통째로 바뀐다.
```
